# PKI Engine Implementation — `admin/tn001`

Offline-root CA pattern. The root CA is generated and kept outside Vault with
OpenSSL. Vault manages only the intermediate CA mounted at `pki-int` inside the
`admin/tn001` namespace. The root CA private key never enters Vault.

## Prerequisites

| Tool | Purpose |
|---|---|
| `vault` | All PKI engine operations |
| `openssl` | Root CA generation, CSR signing, cert inspection |
| `jq` | Parsing Vault JSON responses |

Environment variables required for all `vault` commands in this guide:

```sh
export VAULT_ADDR="https://vault-training-cluster-pegasus-public-vault-f6816f27.65ed0d62.z1.hashicorp.cloud:8200"
export VAULT_TOKEN="<token-with-admin-tn001-rights>"
export VAULT_NAMESPACE="admin/tn001"
```

Setting `VAULT_NAMESPACE` avoids repeating `-namespace=admin/tn001` on every
command. All commands below assume it is set.

> **Token scope:** The token must have policies effective within the
> `admin/tn001` namespace. A root token issued in a parent namespace (e.g.
> `admin`) is valid. A token issued in a sibling or child namespace will not
> traverse upward — ensure the token's accessor namespace matches or the policy
> is applied inside `admin/tn001` directly.

## 1. Offline Root CA

Generate a self-signed RSA-4096 root CA valid for 2 years. Store the key
securely — it never enters Vault.

```sh
mkdir -p .tmp/pki

openssl req -newkey rsa:4096 \
  -x509 \
  -days 730 \
  -noenc \
  -keyout .tmp/pki/root-ca.key \
  -out    .tmp/pki/root-ca.crt \
  -subj   "/CN=tn001 Root CA/O=nhsy-hcp-org/OU=tn001"
```

Inspect:

```sh
openssl x509 -noout -text -in .tmp/pki/root-ca.crt
```

---

## 2. Mount the PKI Secrets Engine

```sh
vault secrets enable \
  -path=pki-int \
  -description="PKI intermediate CA for tn001" \
  -default-lease-ttl=2592000 \
  -max-lease-ttl=31536000 \
  pki
```

Verify:

```sh
vault secrets list
```

---

## 3. Generate the Intermediate CA Key and CSR

Vault generates the private key internally — it is never exported.

```sh
vault write -format=json pki-int/intermediate/generate/internal \
  common_name="tn001 Intermediate CA" \
  key_type=ec \
  key_bits=256 \
  > .tmp/pki/intermediate-response.json

jq -r '.data.csr' .tmp/pki/intermediate-response.json > .tmp/pki/intermediate.csr
```

Writing the full response to a file first avoids a silent empty `.csr` if
the `vault write` command fails mid-pipe. Verify the CSR is non-empty before
proceeding:

```sh
[ -s .tmp/pki/intermediate.csr ] && echo "CSR OK" || echo "ERROR: CSR is empty"
openssl req -noout -text -in .tmp/pki/intermediate.csr
```

The response also contains the `key_id` generated by Vault — capture it for
use in step 7:

```sh
jq -r '.data.key_id' .tmp/pki/intermediate-response.json
```

---

## 4. Sign the Intermediate CSR with the Offline Root CA

Signs the CSR with `pathlen:0` and CA key-usage constraints. Valid for 1 year.

```sh
openssl x509 -req \
  -days 365 \
  -in  .tmp/pki/intermediate.csr \
  -CA  .tmp/pki/root-ca.crt \
  -CAkey .tmp/pki/root-ca.key \
  -CAcreateserial \
  -out .tmp/pki/intermediate-signed.crt \
  -extensions v3_ca \
  -extfile <(printf '[v3_ca]\nbasicConstraints=critical,CA:TRUE,pathlen:0\nkeyUsage=critical,keyCertSign,cRLSign\nsubjectKeyIdentifier=hash\nauthorityKeyIdentifier=keyid:always,issuer')
```

Verify the chain:

```sh
openssl verify -CAfile .tmp/pki/root-ca.crt .tmp/pki/intermediate-signed.crt
```

Expected: `.tmp/pki/intermediate-signed.crt: OK`

---

## 5. Import the Signed Certificate into Vault

```sh
vault write -format=json pki-int/intermediate/set-signed \
  certificate=@.tmp/pki/intermediate-signed.crt \
  > .tmp/pki/set-signed-response.json

cat .tmp/pki/set-signed-response.json | jq .
```

When the intermediate key was generated via `generate/internal` on this same
mount (step 3), Vault automatically links the key to the new issuer and both
`imported_issuers` and `imported_keys` are populated:

```json
{
  "data": {
    "imported_issuers": ["<issuer-id>"],
    "imported_keys":    ["<key-id>"],
    "mapping":          {"<issuer-id>": "<key-id>"}
  }
}
```

If `imported_keys` is empty (e.g. cross-mount import or key generated
externally), the issuer will have no linked key — issue and sign operations
will fail until the key is linked manually in step 7.

## 6. Configure the Cluster and AIA URLs

> **Order matters:** `config/cluster` must be written before `config/urls`
> when `enable_templating=true`. The `{{cluster_path}}` token is resolved
> against the cluster config at write time — if it is absent, the URL strings
> are stored as-is and the tokens will not expand in issued certificates.

Set the cluster path:

```sh
vault write pki-int/config/cluster \
  path="https://vault-training-cluster-pegasus-public-vault-f6816f27.65ed0d62.z1.hashicorp.cloud:8200/v1/admin/tn001/pki-int" \
  aia_path="https://vault-training-cluster-pegasus-public-vault-f6816f27.65ed0d62.z1.hashicorp.cloud:8200/v1/admin/tn001/pki-int"
```

Configure issuing certificate, CRL distribution point, and OCSP URLs with
templating enabled:

```sh
vault write pki-int/config/urls \
  issuing_certificates="{{cluster_path}}/issuer/{{issuer_id}}/der" \
  crl_distribution_points="{{cluster_path}}/issuer/{{issuer_id}}/crl/der" \
  ocsp_servers="{{cluster_path}}/ocsp" \
  enable_templating=true
```

Configure CRL and OCSP behaviour:

```sh
vault write pki-int/config/crl \
  expiry=72h \
  disable=false \
  ocsp_disable=false \
  ocsp_expiry=12h
```

## 7. Set the Default Issuer

List issuers to get the issuer ID imported in step 5:

```sh
vault list pki-int/issuers
```

Read the issuer and confirm key linkage — the `key_id` field must be non-empty.
If it is empty, issue and sign calls will fail with a "no key associated"
error.

```sh
vault read pki-int/issuer/<issuer-id>
```

If `key_id` is empty, link the key manually using the `key_id` captured from
the `generate/internal` response in step 3:

```sh
vault write pki-int/issuer/<issuer-id> \
  key_id=<key-id>
```

Set as the mount default:

```sh
vault write pki-int/config/issuers \
  default=<issuer-id>
```

Optionally set a human-readable name:

```sh
vault write pki-int/issuer/<issuer-id> \
  issuer_name="tn001-intermediate-ca"
```

## 8. Build the CA Chain Bundle (for local verification)

Fetch the intermediate CA cert back from Vault. The standard chain bundle
convention is intermediate first, then root — this is the order TLS servers
and `openssl verify` both expect.

```sh
vault read -field=certificate pki-int/cert/ca > .tmp/pki/intermediate-ca.pem
cat .tmp/pki/intermediate-ca.pem .tmp/pki/root-ca.crt > .tmp/pki/ca-chain.pem
```

> The order here (`intermediate → root`) is correct for TLS server
> configuration. Reversing it (`root → intermediate`) will cause chain
> validation failures in some TLS clients even though `openssl verify` may
> still accept it.

## 9. Create an Issuing Role

Roles define the constraints Vault enforces when issuing certificates.

```sh
vault write pki-int/roles/server \
  issuer_ref="default" \
  allowed_domains="helloworld.example.com" \
  allow_bare_domains=true \
  allow_subdomains=false \
  key_type=ec \
  key_bits=256 \
  ttl=24h \
  max_ttl=24h \
  generate_lease=false \
  no_store=false
```

### Role variable reference

| Field | Type | Default | Description |
|---|---|---|---|
| `issuer_ref` | string | `"default"` | Named issuer to use; `"default"` resolves to the mount default |
| `allowed_domains` | list | `[]` | Domains for which certs may be requested |
| `allow_bare_domains` | bool | `false` | Permit the bare domain (e.g. `example.com`) |
| `allow_subdomains` | bool | `false` | Permit subdomains (e.g. `*.example.com`) |
| `key_type` | string | `"ec"` | `ec`, `rsa`, or `any` |
| `key_bits` | number | `256` | Key size — 256 for EC P-256; 2048/4096 for RSA |
| `ttl` | string | `"24h"` | Default cert TTL when not specified at request time |
| `max_ttl` | string | `"168h"` | Maximum cert TTL this role can issue |
| `generate_lease` | bool | `false` | Whether Vault creates a lease for issued certs |
| `no_store` | bool | `false` | Skip storing the cert in Vault storage |

List and read roles:

```sh
vault list pki-int/roles
vault read pki-int/roles/server
```

## 10. Issue a Certificate (Vault generates key + cert)

Vault generates the private key. The key is returned in the response and never
stored — retrieve it immediately.

```sh
vault write -format=json pki-int/issue/server \
  common_name=helloworld.example.com \
  ttl=1h \
  > /tmp/issue.json
```

Extract the certificate, private key, and chain:

```sh
jq -r '.data.certificate'          /tmp/issue.json > /tmp/issued.pem
jq -r '.data.private_key'          /tmp/issue.json > /tmp/issued.key
jq -r '.data.issuing_ca'           /tmp/issue.json > /tmp/issuing-ca.pem
jq -r '.data.ca_chain | join("\n")' /tmp/issue.json > /tmp/ca-chain.pem
```

> `jq -r '.data.ca_chain | join("\n")'` ensures a newline between PEM blocks.
> Using `.data.ca_chain[]` without `join` concatenates blocks without
> separators, which breaks some PEM parsers.

Verify chain and inspect:

```sh
openssl verify -CAfile .tmp/pki/ca-chain.pem /tmp/issued.pem
openssl x509 -noout -subject -issuer -dates -in /tmp/issued.pem
```

## 11. Sign a Client-Provided CSR

The client generates and retains the private key. Vault signs only the CSR.

Generate a throwaway EC P-256 key and CSR:

```sh
openssl req -newkey ec -pkeyopt ec_paramgen_curve:P-256 -nodes \
  -keyout /tmp/client.key \
  -out    /tmp/client.csr \
  -subj   "/CN=helloworld.example.com"
```

Submit the CSR to Vault:

```sh
vault write -format=json pki-int/sign/server \
  csr=@/tmp/client.csr \
  common_name=helloworld.example.com \
  ttl=1h \
  > /tmp/sign.json

jq -r '.data.certificate' /tmp/sign.json > /tmp/signed.pem
```

Verify chain and confirm key matches the certificate:

```sh
openssl verify -CAfile .tmp/pki/ca-chain.pem /tmp/signed.pem

cert_pub=$(openssl x509 -noout -pubkey -in /tmp/signed.pem)
key_pub=$(openssl pkey -pubout -in /tmp/client.key)
[ "${cert_pub}" = "${key_pub}" ] && echo "key/cert match" || echo "MISMATCH"
```

## 12. Revoke a Certificate

Revoke by serial number. The serial is returned in the issue/sign response
under `.data.serial_number`.

```sh
vault write pki-int/revoke \
  serial_number=<serial-number>
```

Force an immediate CRL rebuild after revocation (otherwise the CRL refreshes
on its configured `expiry` schedule):

```sh
vault write pki-int/crl/rotate
```

Verify the serial appears in the updated CRL:

```sh
vault read -field=certificate pki-int/cert/crl | openssl crl -noout -text \
  | grep -A2 "Revoked"
```

## 13. PKI Storage Tidy

After revocations or intermediate CA rotation, tidy removes expired certificates
and CRL entries from Vault storage. Run after bulk revocations or after
importing a new intermediate.

```sh
vault write pki-int/tidy \
  tidy_cert_store=true \
  tidy_revoked_certs=true \
  safety_buffer=72h
```

`safety_buffer` sets how long past expiry a cert must be before tidy removes
it. Check tidy status:

```sh
vault read pki-int/tidy-status
```

## 14. Inspecting the PKI Mount

Read current URL configuration:

```sh
vault read pki-int/config/urls
vault read pki-int/config/crl
vault read pki-int/config/cluster
```

Inspect the intermediate CA certificate:

```sh
vault read -field=certificate pki-int/cert/ca \
  | openssl x509 -noout -text
```

Inspect AIA, CRL, and OCSP extensions in an issued cert:

```sh
openssl x509 -noout -text -in /tmp/issued.pem \
  | grep -A5 "Authority Information\|CRL Distribution"
```

Fetch and inspect the CRL:

```sh
vault read -field=certificate pki-int/cert/crl | openssl crl -noout -text
```

List all issuers and read a specific one:

```sh
vault list pki-int/issuers
vault read pki-int/issuer/<issuer-id>
```

## 15. Recovery — Intermediate Mount Recreated

If the `pki-int` mount is destroyed and re-created, the intermediate key and
CSR are gone. Repeat the full intermediate CA lifecycle:

1. Re-enable the PKI mount (step 2)
2. Generate a new internal key and CSR (step 3)
3. Sign the new CSR with the offline root CA (step 4)
4. Import the newly signed certificate (step 5)
5. Re-configure cluster path, URLs, and CRL config (step 6)
6. Set the new default issuer (step 7)
7. Rebuild the local CA chain bundle (step 8)
8. Re-create roles (step 9)

> The root CA key (`root-ca.key`) and cert (`root-ca.crt`) are reused across
> intermediate CA rotations — do not regenerate the root unless you intend to
> rotate the entire trust anchor.

## 16. ACL Policy Reference

Minimum capabilities required inside the `admin/tn001` namespace:

```hcl
path "sys/mounts/pki-int" {
  capabilities = ["create", "read", "update", "delete", "list", "sudo"]
}

path "sys/mounts/*" {
  capabilities = ["create", "read", "update", "delete", "list", "sudo"]
}

path "pki-int/*" {
  capabilities = ["create", "read", "update", "delete", "list", "sudo"]
}

path "pki-int" {
  capabilities = ["read", "list"]
}
```

For issue/sign-only consumers (application tokens), scope to the specific role:

```hcl
path "pki-int/issue/server" {
  capabilities = ["create", "update"]
}

path "pki-int/sign/server" {
  capabilities = ["create", "update"]
}
```
